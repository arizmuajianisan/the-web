## The pain we‚Äôre solving

Manually bumping versions, writing changelogs, and cutting GitHub releases is tedious and error-prone.  
With `@arizmuajianisan/gitworkflow`, [semantic-release](https://semantic-release.gitbook.io) does all of that for you‚Äîbased solely on the semantics of your commit messages.

## Install & configure in 30 seconds

### Quick (no install)
```bash
npx @arizmuajianisan/gitworkflow
```

You can use `--yes` to skip the wizard. By default, it will install Prettier for code formatting. And need to add the `GH_TOKEN` to `.env` manually.

### Global install
```bash
npm i -g @arizmuajianisan/gitworkflow
npx gitworkflow
```

The CLI scaffolds:

- commitlint + husky pre-commit hooks  
- semantic-release config  
- GitHub Actions workflow (releases on push to `main`)
- .env file with GH_TOKEN

Done!‚Äîyour repository is now release-ready üöÄ.

After the wizard finishes, add a classic GitHub token with `repo` scope to `.env`:

```bash
echo "GH_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx" > .env
```

Done!‚Äîyour repository is now release-ready üöÄ.

## Writing commits the robot understands

Every commit is linted before it‚Äôs accepted. Format:

```markdown
<type>(<scope>): <Imperative subject>
```

Examples:

```bash
feat(auth): Add passwordless login
fix(api): Return 404 for missing users
docs(readme): Update installation steps
```

Breaking changes go in the footer:

```bash
feat(auth): Add passwordless login

BREAKING CHANGE: the /login endpoint now requires a JWT
```

When this commit lands on main, semantic-release will:
- bump the correct major/minor/patch version
- update CHANGELOG.md
- create a GitHub release with assets
- publish to npm (if package.json says so)

Or, you can simply ad the `!` after type to create a major release:

```bash
feat!: Add passwordless login
```

> **Note**: To make the `!` work, you need to use a single-quote when committing. Example:

```bash
'feat!: Add passwordless login'
```

## What happens under the hood

1. husky installs Git hooks.
2. commitlint validates the message.
3. On push, release-it runs semantic-release.
4. release-it analyzes commits, decides version, writes changelog, tags, releases, and publishes.

## Customising the rules

Edit `.commitlintrc.json` to relax or extend conventions:

```js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'build', 'chore', 'ci', 'docs', 'feat', 'fix',
      'perf', 'refactor', 'revert', 'style', 'test'
    ]],
    'type-case': [2, 'always', 'lower-case'],
    'subject-case': [2, 'always', 'sentence-case'],
    'header-max-length': [2, 'always', 100]
  }
};
```

> By default, in the subject case, it uses `sentence-case`, but you can change it to `lower-case` or `upper-case`. This is my personal preference.

Commit, push, and your next release will respect the new rules.

## Troubleshooting corner

| Symptom                           | Quick fix                                                                                                                                      |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| ‚ÄúNo GH\_TOKEN‚Äù error on CI        | Ensure `.env` is *not* git-ignored locally, but set the token as a **repository secret** (`Settings ‚Üí Secrets ‚Üí GH_TOKEN`) for GitHub Actions. |
| Commits rejected with ‚Äútype-enum‚Äù | Check allowed types in `.commitlintrc.json`.                                                                                                   |
| Release not created after push    | Make sure `main` branch is the default and the workflow file is on `main`.                                                                     |
